## Executive Summary *(한 장 요약)*

**우리가 만들 것.** 한국 콜센터·리드 영업 조직을 위한 멀티테넌트 SaaS CRM. 랜딩페이지 수집 → 자동 분배 → 상담자 업무 → 매출 전환까지를 **끊김 없이** 처리한다.

**왜 지금.** 시장 1위 선택지인 **디비카트·1000.sseye**가 2006년부터 큰 변화 없이 운영되며 기능 갭이 크다. 대표는 디비카트 사용 중 NOTI 전송 실패로 약 **2천만 원 손실**을 직접 경험했다 — 같은 고통을 겪는 광고주가 전국에 많다. tmrw_web 랜딩 서비스 기존 고객을 그대로 업셀 가능해 초기 콜드 세일즈 리스크가 낮다.

**핵심 차별화 (한 줄씩).**
1. **절대 안 놓친다** — NOTI 실패 실시간 빨간불 (이미 tmrw_web에서 검증됨)
2. **실제 상담을 돕는다** — 통화 이력·콜백·문자·AI 요약 (경쟁사 전부 없음)
3. **현대적 UX** — 반응형 모바일·키보드 단축키·실시간 업데이트 (2026년 수준)
4. **랜딩+CRM 통합** — tmrw 생태계 안에서 한 계정
5. **리드 품질 관리** — 중복·블랙리스트·스코어링

**수익 모델.** 회사 단위 월 구독. Starter 5만 / Pro 15만 / Enterprise 50만+ (안 — 상세는 §10).

**MVP 목표 *(3개월 후 = 2026-07월 말).***  베타 10개 고객사 온보딩 → **9월 말 정식 런칭 → 연말까지 30개사 / MRR 1천만 원**.

**결정 요청 사항 (§11).**  (1) 방향 승인 (2) 기술 스택 (3) 출시일 (4) 인력/예산 (5) 브랜드명.

## 1. 시장 분석

### 1.1 경쟁 제품 현황 (2026-04 기준)

| 제품 | 강점 | 약점 | 가격 (추정) |
|---|---|---|---|
| **디비카트** | 시장 인지도, 광고주 풀 | NOTI 3회 재시도 후 포기 → **실제 손실 발생**, 낡은 UI | 월 10-30만 |
| **1000.sseye** | 자동 분배 엔진 견고 | 상담 기능 거의 없음(메모만), 2006년 UI, 모바일 X | 월 10-30만 |

### 1.3 수요 근거

- tmrw_web 기존 랜딩 고객 (현재 N개사) → 자연스러운 CRM 업셀 경로
- 디비카트 손실 경험 공감대 — 대표님 포함 광고주 커뮤니티에서 반복 제기
- 금융·성형·학원·부동산 업종은 콜센터 기반이라 CRM 의존도 높음

## 2. 제품 비전

> **"리드를 1건도 잃어버리지 않고, 최소 인원으로 최대 매출을 만드는 한국형 CRM."**

### 핵심 3대 가치

1. **안 놓친다 (Reliability)** — 수집부터 상담 완료까지 실패 건을 실시간 감지/알림
2. **실제 업무를 돕는다 (Workflow)** — 통화·메모·상태·SMS·AI 요약이 한 화면에
3. **현대적이다 (Modernity)** — 2026년 기준 모바일/반응형/실시간/키보드 퍼스트

## 5. 차별화 포인트 (상세)

### 5.1 NOTI 실패 빨간불 모니터링 🔴

**우리가 이미 검증한 기능** (tmrw_web 프로덕션에서 운용 중). 외부 시트/API로 NOTI 나갈 때 실패 시:
- 폼 소유자에게 SMS 즉시 발송 (10분 throttle, SMS 폭탄 방지)
- 리드 리스트 비고 컬럼에 🔴 표시, 담당자가 수기 조치 후 수동 해제

**경쟁사 대비 의미.** 디비카트 loss 사례의 **직접 해결책**. 영업 단계에서 가장 강력한 USP.

### 5.2 상담 타임라인

리드 상세 화면에 아래가 한 눈에:
```
⏱ 2026-04-22 14:03  [접수]   강팀장이 A사 랜딩으로 접수
⏱ 2026-04-22 14:05  [자동분배] 테스터3에게 분배
📞 2026-04-22 15:20  [통화]   12분 통화 (녹음 있음)
📝 2026-04-22 15:32  [메모]   "담보대출 관심, 금요일 재통화 요청"
🔔 2026-04-26 10:00  [콜백 예약]
📱 2026-04-22 15:33  [SMS]   "안녕하세요 테스터3 상담사입니다..."
```

### 5.3 현대적 UX

- **모바일 반응형.** 상담자가 이동 중에도 리드 확인 가능 (sseye 모바일 X)
- **키보드 단축키.** `j/k` 리드 이동, `e` 편집, `s` 상태전환, `m` 메모 추가
- **실시간 업데이트.** Firestore snapshot listener — 새 리드 들어오면 테이블에 즉시 반영
- **일괄 작업.** 체크박스 → 배치 상태 변경 / 일괄 재배정 / 일괄 삭제
- **다크 모드.** 장시간 근무 상담자 피로 감소

### 5.4 AI 통합 (Phase 3 홍보 앵커)

- Whisper로 통화 녹음 자동 텍스트화
- Claude로 핵심 요약 + 다음 액션 추천 (예: "고객이 금요일 재통화 요청함, 금리 비교 자료 준비 필요")
- 영업 문서에 "AI-native CRM"으로 포지셔닝 가능

### 5.5 리드 품질 관리

- **중복 전화번호 경고.** 같은 번호 다른 랜딩으로 재제출 시 최초 건과 연결 or 경고
- **블랙리스트.** 어뷰징/반복 이탈자 자동 필터
- **IP·기기 중복 감지.** 동일 IP에서 단시간 다수 제출 시 플래그

## 6. 데이터 모델 초안 (Firestore)

```
companies/{cid}                      — SaaS 테넌트 최상위, 절대 격리 단위
├── info                              (회사명, 요금제, 결제 정보, 사용량)
├── users/{uid}                       — role: master | accept | consult
│   ├── username, login_id, phone, email
│   ├── manager_id, ancestors[]       — 조직 계층 (tmrw_web 패턴 재사용)
│   └── distribution_quota            — 자동분배 일일 쿼터
├── landings/{lid}                    — 광고 분류 = 리드 소스
│   ├── title, api_token
│   └── field_map                      — ITEM/DBname/DBphone/DBfild1~N 매핑
├── fields/{fid}                      — 커스텀 필드 정의
│   ├── title, type (select|text), options[]
│   └── sort_order
├── leads/{lead_id}
│   ├── receiver_uid, counselor_uid, landing_id
│   ├── status (접수|상담중|선별중|진행중|결과)
│   ├── distribution_type (auto|manual|none)
│   ├── created_at, received_at, ip, user_agent
│   ├── name, phone, fields{...}
│   ├── noti_status (ok|failed|failed_cleared)
│   └── timeline/{entry_id}           — 통화·메모·상태전환·SMS 로그
│       └── type, actor_uid, at, payload{...}
├── distribution_config
│   └── auto_on, weekdays[], time_start, time_end
├── blacklist/{phone}                  — 전화번호 차단 목록
├── templates/{tid}                    — SMS·알림톡 템플릿
└── stats_daily/{yyyymmdd}             — 집계 카운터

public_api/
  POST /api/v1/lead                   — 외부 → CRM 리드 수신
  POST /api/legacy?ITEM=&DBname=...   — 디비카트 호환 포트
```

**핵심 설계 결정.**
- 회사별 최상위 분리 → 데이터 격리 자동, 보안 감사 단순화
- `leads/{lead_id}/timeline/*` 서브컬렉션 → 리드별 활동 로그 무한 확장
- 디비카트 호환 레거시 API 포트 유지 → 기존 광고주 migration 마찰 0

## 7. 기술 스택 제안

| 레이어 | 후보 | 추천 | 근거 |
|---|---|---|---|
| Backend | Flask+Firestore / FastAPI+Postgres / Node+Postgres | **Flask + Firestore** | tmrw_web과 일관 → 학습·운영 비용 절감, 빠른 개발 |
| Frontend | 바닐라 JS / Next.js / SvelteKit | **Next.js (React)** | 현대적, 반응형, SSR, 생태계 방대 |
| Realtime | Firestore listener / WebSocket | **Firestore listener** | SDK 내장, 무료 웹소켓 대체 |
| Infra | GAE Standard / Cloud Run / Vercel | **GAE(API) + Vercel(FE)** | GAE는 이미 사용, Vercel은 Next.js 최적 |
| SMS·알림톡 | NCP SENS | 유지 | tmrw_web 이미 연동 |
| 녹취 저장 | GCS (암호화) | 유지 | 저렴, 한국 리전 지원 |
| AI | Claude / OpenAI | **Claude** | 한국어 품질, 장기 문맥 (100만 토큰) |
| 결제 | Toss Payments / Stripe | **Toss Payments** | 한국 B2B 정기결제 최적 |
| 모니터링 | GCP Cloud Logging / Sentry | 둘 다 | GCP 기본 + Sentry 프론트 에러 |

### 리스크 & 대안

- **Firestore 비용.** 리드당 월 read 수백 회 예상. 고객사 많아지면 월 수백만 원 돌파 가능. 대책: (1) 캐싱 적극 도입 (2) 조기부터 비용 모니터링 (3) Postgres 이전 경로 열어둠
- **Firestore 쿼리 한계.** 복합 인덱스 필요한 쿼리 많음. 사전에 index.yaml 체계적으로 관리
- **Vercel 분리 구조 복잡도.** CORS/인증 쿠키 이슈 — SameSite/Domain 설정 초기부터 확정
